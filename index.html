Chapter 01

話し手について — ギプスの 2 ヶ月で学んだこと

Eric 氏は Anthropic のリサーチャーで、コーディングエージェントの領域を専門としている。Barry Zhang 氏との共著論文「Building Effective Agents」で、Anthropic としてのエージェント設計のサイエンスとベストプラクティスを世に出した人物でもある。

しかし今回のトークのきっかけは、もっと個人的な出来事だった。昨年、自転車通勤中に手を骨折し、2 ヶ月間ギプス生活になった。そのあいだ、彼のコードは——自分の手では一行も書かれなかった。

そのギプスの 2 ヶ月間、僕のコードはすべて Claude が書いていた。 Eric

「どうすればこれを効果的に回せるか」を本気で考えざるを得ない状況に置かれたのだ。そこでの学びは、やがて Anthropic の他プロダクトやモデル本体にも流れ込んでいった。今日のトークは、その 2 ヶ月と、その先にある「本番でのバイブコーディング」の話である。

なぜこの話が面白いのか

制約は、思考を深くする。怪我という偶然が、Eric 氏を「AI にコードを任せるとはどういうことか」を最前線で考えざるを得ない立場に押し出した。この講演の実感のこもった言葉の源泉は、その 2 ヶ月にある。

Chapter 02

バイブコーディングとは — 定義を厳密にしておく

まず、言葉の定義から揃えよう。世の中には「AI を大量に使ってコードを書いてもらう＝バイブコーディング」と雑に同一視している人が多い。Eric 氏はそれは違うと言う。

Cursor や Copilot を使って、AI が生成するコードを一行ずつ読んで取捨選択している——これはまだタイトなフィードバックループの中にいる。コードを書いているのは人間で、AI は伴走者にすぎない。

Vibe coding is when you fully give into the vibes,
embrace exponentials, and forget that the code even exists. Andrej Karpathy · バイブコーディングの原義

Andrej Karpathy 氏のこの定義の核心は、最後の一節——「コードが存在することすら忘れる」——にある。Eric 氏が言う「バイブコーディング」も、常にこの意味で使われる。

なぜ世の中がざわついたのか

バイブコーディングが注目を浴びたのは、エンジニアではない人々がコード生成に熱狂しはじめた瞬間だったからだ。Copilot や Cursor は結局エンジニア向けのツールだった。だがバイブコーディングは、コードを書いたことのない人でも、一人でアプリ全体を作り上げる扉を開いてしまった。

これは多くの人にとって巨大な解放だった——そしてもちろん、巨大な副作用も伴った。

Chapter 03

落とし穴と期待値 — なぜ「おもちゃ」にしか見えないのか

副作用の代表例は、SNS をにぎわせた数々の事故である：

  • API キーの使用量がマックスに張り付く——コード上の不備で際限なく呼び出される
  • サブスクリプションの課金バイパス——認証や支払いロジックが穴だらけ
  • DB に得体の知れないデータが生成される——スキーマも権限も曖昧なまま突き進む

これらはすべて「自分が今何をしているのか分かっていない人」がプロダクションレベルのものを作ろうとしたときに起こる。

ポジティブ事例は、たいてい低リスク

逆に、バイブコーディングがうまく機能している例はどうかというと、ほぼすべてが「低リスクな領域」だ。自作のビデオゲーム、一晩で作るサイドプロジェクト、バグが出ても笑い話で済む遊びの類。

ここで多くの人が思う素朴な疑問

実プロダクトに使うとリスクが跳ね上がり、うまくいっているのはオモチャばかり——なら、バイブコーディングなんて気にしなくていいのでは？

Eric 氏の答えは一言だ——「指数関数がある限り、気にするしかない」。次章で、この答えを真正面から解剖する。

Chapter 04

指数関数の波 — 7 ヶ月で倍になる世界

AI がこなせるタスクの長さ（一度の指示で完結できる作業量）は、約 7 ヶ月ごとに倍になっている。2025 年の現在地は「およそ 1 時間分の仕事」。

今はまだバイブコーディングを無理にしなくていい。Cursor に任せ、Claude Code に機能を書かせ、出てきたコードをレビューする——そのやり方で十分やっていける。人間が密にループに関われる規模だからだ。

しかし、来年は？ 再来年は？

1 時間が 2 時間になり、4 時間になり、1 日分の仕事、1 週間分の仕事をモデルが一息に吐き出すようになる。そのとき、コードを一行ずつ読むワークフローは物理的に追いつかない。

指数関数の波に乗りたいなら、どこかで「責任を持って手放す」方法を見つけなければならない。 Eric
図 1 — タスク長の指数関数。「今はまだ」読める。「来年は」読みきれない。

コンパイラの喩え — 歴史は繰り返す

Eric 氏が好む喩えがある。コンパイラ黎明期——当時のエンジニアの多くは、コンパイラを信用していなかった。生成されたアセンブリを目で読み、「自分が書くのと同じようになっているか」を毎回確認していた。

だが、システムが大きくなるにつれ、そのやり方は物理的に成立しなくなった。どこかで「信じる」しかなくなる。

The code exists, but you no longer need to look at it. コードは存在する、ただし読まなくていい。 コンパイラがもたらしたパラダイム転換

僕らのソフトウェア業界全体への挑戦状は、ここにある——「プロダクトが存在することは忘れずに、コードが存在することを忘れる」。アセンブリは今もそこにある。だがほとんどの開発者は、もうアセンブリを読まずに良いソフトウェアを作れている。同じことを、もう一段上のレイヤーで起こす。

古くて新しい悩み — 実は古典的マネジメント問題

ここで Eric 氏は、ソフトウェア業界の外に目を向ける。この問題は実は、文明史レベルで繰り返されてきた古い問題だというのだ。

  • CTOはその専門領域の一線の現場よりコードが書けるとは限らない。が、優秀な専門家の成果物を評価する方法を持っている。
  • PMは、完成した機能のコードを読まない。だがプロダクトを使ってみて、意図通りに動くかを確かめられる。
  • CEOは会計の専門家ではない。それでも分かるキーファクトとデータのスライスを抜き打ちチェックすることで、全体財務モデルの健全性に自信を持てる。

つまり「自分が実装を理解していないものを、どう管理するか」は、世の中の全マネージャーが日常的に解いてきた問題である。ソフトウェアエンジニアだけが、これまで純粋な個人コントリビューターとしてスタック全段を自分で理解することに慣れすぎていたのだ。

核心

実装を知らずとも検証可能な「抽象レイヤー」を見つけろ。これが、責任あるバイブコーディングの一行サマリである。

Chapter 05

葉ノード戦略 — 技術的負債を「押し込める」場所

ただし、Eric 氏は一つだけ正直な but を付け加える。技術的負債（tech debt）だけは、今のところコードを読まずに検証する良い方法がない。

会計は監査で検証できる。プロダクトは使って検証できる。ところが技術的負債は、実装自体の専門家でないと見抜けない——これは現時点での厳然たる制約である。

戦略：葉ノードに集中する

この制約を認めた上で、それでもバイブコーディングを本番に取り入れる方法がある。答えは、コードベースの「葉ノード」に集中すること。

図 2 — 葉ノードに技術的負債があっても波及しない。幹は深く理解せよ。

なぜ葉ノードなら OK なのか

  • 何も依存していない——末端機能なので、負債が周囲に波及しない
  • 変更される可能性が低い——端の装飾、末端の機能は、書き換えが少ない
  • 上に何かが積み上がらない——別の機能の土台にはならない

逆に幹や太い枝——中核アーキテクチャ、共有インフラ、横断ユーティリティ——は、人間が今後も深く理解し続ける必要がある。そこにこそ他の機能が積み上がり、そこが拡張可能で読みやすく柔軟であることが、組織全体の速度を決める。

朗報：境界は動く

モデルは毎月賢くなっている。Eric 氏は「この 1〜2 週間 Claude 4 を使って、3.7 の頃よりずっと信頼できるようになった」と語る。「葉ノードにしか使えない」という境界は、年々内側（幹側）へ侵食していく。

Chapter 06

Claude の PM になる — 成功の核

バイブコーディングで成功するための、Eric 氏のたった一つのアドバイス——

Ask not what Claude can do for you,
but what you can do for Claude. Eric · JFK へのオマージュとして

バイブコーディング中、あなたの役割は「Claude のプロダクトマネージャー」である。PM のように考えよ：このタスクで成功するために、新入社員ならどんなガイダンスとコンテキストが要るか？

コンテキストの集め方 — 15〜20 分を投資する

多くの人は、AI に対して短いキャッチボール型のやり取りをしがちだ：「この機能作って」「このバグ直して」。しかし、もし出社初日の新入社員に「この機能を今すぐ実装して」とだけ言ったら？——成功できる見込みは、ほぼゼロだ。

Eric 氏は、機能を Claude に任せるとき最初の 15〜20 分をコンテキスト集めに使うという。しかも、その時間は「人間が一人でプロンプトを書いている時間」ではない：

  • 別セッションで Claude と会話する。コードベースを探検させ、関連ファイルを探させる
  • 一緒に計画を練る——何をしたいか、どのファイルを触るか、既存のどのパターンに従うべきか
  • その対話の結晶として、一つのプロンプト（または計画書）を作る
  • それを携え、別の新しいコンテキストに移って、「この計画を実行して」と指示

この手間を前払いすると、Claude の成功率は劇的に跳ね上がる。逆にここを省略すると、後段で何度も手戻りする。

正しい問いを立てる — 全員がバイブコードすべきとは限らない

ここで Eric 氏は、トークのタイトルに反する正直な告白をする。

I don't think that vibe coding in prod is for everybody. 本番でのバイブコーディングは、全員に向いているわけではない。 Eric

完全に非技術的な人がゼロからビジネスを構築するのは危険だ、と彼は言う。なぜか——正しい問いを立てられないからだ。PM が仕様を詰められないのと同じで、Claude の PM が Claude に効果的な指示を出せない。

線の引き方

ゲームや創作、個人の道具作りにおけるバイブコーディングは素晴らしい。しかし本番のプロダクションシステムには、「正しい問いを立てられる」だけの技術的理解が必要。ここは区別すべき、と Eric 氏は明確に線を引く。

Chapter 07

2 万 2 千行の PR — Anthropic 社内の実例

抽象論はここまで。Eric 氏は、Anthropic 社内で最近起きた生々しい実例を紹介する。

事実

Anthropic の本番強化学習（RL）コードベースに、22,000 行にわたる変更を含む PR がマージされた。このコードの大部分は、Claude が書いた。

ではどうやって、この規模を責任を持ってねじ込めたのか。Eric 氏は 4 つの要素を挙げる：

PILLAR 01

Claude の PM に徹した

単発プロンプトでマージしたのではない。要件定義、Claude への誘導、システム設計——数日分の人間の仕事が前段にあった。PM ロールを全員が自覚的に引き受けた。

PILLAR 02

変更を葉ノードに集中

近い将来、変更が入らないと分かっている末端機能に意図的に集中。「多少の負債があってもこの場所なら許容できる」ラインを引いた。

PILLAR 03

幹部分は人間レビュー

拡張性が重要になる部分だけ、徹底的に人間が読んだ。メリハリのついたレビュー配分。

PILLAR 04

検証可能な設計

入力・出力を人間が確認しやすい形に意図的に設計。ストレステストを長時間走らせて安定性を実測。「読まなくても確かめられる」チェックポイントを仕込んだ。

結果 — 自信は「手書きレビュー」と同等

結論として、この PR に対する信頼感は、普段の変更と同等まで持ち上がった。しかも所要時間は手で書いてすべてをレビューする場合のほんの一部。

そしてもう一つ、Eric 氏が興奮を隠さず語ったのは——

これが一番ワクワクしたのは、1 週間を節約したこと自体より、
「これができる」と知ったあとに、考え方そのものが変わったことだ。 Eric

以前は「2 週間かかるから今回は見送ろう」としていた機能が、1 日でできるなら、やっちゃおうになる。ソフトウェアの限界費用が下がり、消費と創造の容量が一段上がる——そこが本当の恩恵だと彼は言う。

Chapter 08

持ち帰るべきこと — 4 つのクロージング

Eric 氏は、トークの最後に、明日から使える 4 つの原則に凝縮する。

  1. Be Claude's PM.

    Claude の PM になりきること。コンテキスト、要件、制約を、新入社員に説明するつもりで用意する。

  2. Focus on leaf nodes.

    バイブコーディングは葉ノードに。中核アーキテクチャと基盤システムは人間の領域として守る。

  3. Think about verifiability.

    「コードを読まずに正しさをどう確かめるか？」——これを設計段階から問い続ける。入出力を検証可能に、チェックポイントを仕込む。

  4. Remember the exponential.

    今日はまだバイブコーディングをしなくていい。1〜2 年後には、それが大きなディスアドバンテージになる。モデルが 1 週間分の仕事を出力する世界で、「全行読む／全行書く」を掲げる人は、ボトルネックになる。

Eric からのメッセージ

これは、今後数年のソフトウェア産業にとって最大のチャレンジの一つになる。——だからこそ、今のうちに手触りを獲得しておこう。

Chapter 09

Q&A — 会場からの質問

かつては構文エラーやライブラリ連携で「苦労して」学んだ。今後、良いバイブコーダー／良い PM になるには？

心配な面と楽観的な面、両方あります。心配な面は、「苦労の中でしか学べないもの」に手が届きにくくなること。とはいえ、大学時代の教授が「今の若い奴らはアセンブリを手で書いたことがないから本当には速いコードが書けない」と嘆いていたのと、同じ種類の変化でもあります。

楽観的な面は、僕自身、AI ツールを使うと圧倒的に速く新しいことを学べること。Claude と一緒にコーディングしている時、「このライブラリ見たことない、教えて。なぜこれを選んだの？他と比べてどう違う？」と聞けば、常駐のペアプログラマーとして答えが返ってくる。

怠け者は学ばなくなる。でも学びたい人にとっては、これほど豊かな時代はない。Claude は、自分がバイブコードで作ったものの中身を一緒に読み解いてくれる。

もう一つの面——「良いアーキテクチャだったか」が分かるサイクルが、今まで 2 年かかっていたものが 6 ヶ月に縮む可能性がある。本気で学ぼうとするエンジニアは、同じ暦期間で4 倍の数のレッスンを吸収できる。

プリプラニングで「情報を与えすぎ」と「足りない」のバランスは？ PRD のような定型テンプレを使っている？

何を大事にしているかで変わります。実装の詳細にこだわらないタスクなら、「要件はこれ、最終状態はこれ」だけ書いて、実装方針には触れない。一方、コードベースを熟知している領域では、「このクラスを使って、この類似機能を参考に」とかなり踏み込む。

全体としては——過剰に制約しない方が、モデルは良い仕事をする。厳格なテンプレ運用にそれほどこだわる必要はない。新入社員に渡すつもりで——それくらいの温度感で十分です。

セキュリティとの両立は？ 先日、有名バイブコードアプリ 10 個がハッキングされた事件もあった。

これはまさに「Claude の PM」の話に戻ります。PM として「どこが危険か、どこが安全か、どこを注意深く扱うべきか」を十分に理解していること。

メディアで騒がれているバイブコーディング事故の多くは、本来、本番システムを書くべきではない人が書いてしまったケース。言い方が悪かったので言い直すと——重要な本番システムをバイブコードすべきではない層がやってしまっているケース、です。

Anthropic の先の 2 万 2 千行の例は、完全にオフライン環境で動くシステムだった。だからこそセキュリティ面で「絶対に事故は起こり得ない」という高い確信を持てた。バイブコーディングの適用範囲は、状況と文脈で判断するのが王道です。

世界人口の 0.5% 未満がソフトウェア開発者。ノンエンジニアがもっと安全にバイブコードで構築できるようになるには、プロダクトはどう変わるべき？

「証明可能に正しい」フレームワークやプロダクトがたくさん生まれてほしい。例えば——バックエンドで危ない部分（決済、認証）は既製で組み込み済み、ユーザーはUI レイヤーだけを埋めるような、空欄穴埋め型のサンドボックス。

最もシンプルな先行例は Claude Artifacts。認証も決済もない、フロントエンドのみの環境だから、どれだけ乱暴にコードを生成しても安全。この発想をバックエンド付きで、どんな乱戦が起きても壊れないホスティング環境として実装したプロダクトがあれば、それは面白いビジネスになる。

バイブコーディングと補完関係にあるツールを誰か作ってください——僕たちもそれに期待しています。

テスト駆動開発のコツは？ 実装を先に書かせるとテストが後付けになりがち。

TDD はバイブコーディングでとても有効です。テストケースが理解できる範囲なら、それ自体が検証の抽象レイヤーになるから。

Claude は放っておくと「実装に密着しすぎたテスト」に寄りがち。僕はかなり明示的に「エンドトゥエンドのテストを 3 つ書いて。ハッピーパス、エラーケース A、エラーケース B。全体で 3 つだけ」と指定します。このくらい規範的な方が、テストが「自分が読んで分かる抽象レイヤー」に留まってくれる。

バイブコーディング時、コード本体より先にテストだけは必ず読む——テストに合意できて、それが通っているなら、実装への信頼度は一気に上がる。

「指数関数を受け入れろ」とは具体的にどういうこと？ モデルは改善するとはいえ、全軸で同じように伸びるわけでは？

鋭い質問です。指数関数の本質は「毎年良くなる」という話ではなく——想像できる速度より速く良くなるという話なんです。ある点を超えると、ただ線形に増えているようには見えなくなる。ワイルドな領域に一気に行く。

Dario（CEO）と Mike Krieger（CPO）のトークに印象的な言葉がある——「Machines of Loving Grace はサイエンスフィクションではない。プロダクトロードマップだ」。遠い未来の話に聞こえるけど、指数曲線に乗っていると、想像の外側へ高速で進む。

90 年代のコンピュータに関わっていた人に「RAM が数 KB しかない」と言われ、「はい、今は数 KB、来年は数 KB 増えます」と答えたとしたら——現在のテラバイトという答えは出てこない。2 倍どころか、百万倍になった。

「指数関数を受け入れる」とは、20 年後のモデルは今より 2 倍賢いという予想ではなく、今より百万倍賢く、百万倍速い世界を前提に考え始めることです。それは正直、どういう世界なのか、僕らにはまだちゃんと想像できません。

ターミナルの Claude Code 派と VS Code/Cursor 派、どちらを使う？ Claude Code の /compact はどれくらいの頻度で？

両方使います。VS Code の隣でターミナル Claude Code を開き、Claude Code に編集させつつ、VS Code 側で進行を眺める——この形が多い。純粋なバイブコーディングというより、テストだけ読むくらいのハイブリッドです。

/compact は、人間の開発者なら「ここで昼休憩入れたいな」と思う区切りで使います。たとえば「関連ファイルを全部探させてプランを作らせた」あと、そのプランをドキュメントに書き出させてから compact する。これで、計画立案に使った 100K トークンが、数千トークンの抽出結果に圧縮される。

セッションを清潔に保つコツは、「良いストッピングポイントで区切る」ことです。

複数の Claude Code を走らせたり、Git ワークツリーを使う運用は？ 不慣れなコードベースで素早く PR を出すコツは？

複数ツールの併用——Claude Code で始めて、細かい修正は Cursor で直接やる、というスタイル。具体的な一行を書き換えたいときは、僕は Cursor の方がストレートに届く。

不慣れなコードベースへの入り方は、機能実装の前に、Claude Code で地図作りをします：「認証はこのコードベースのどこで起きている？」「これと似た機能はどこにある？」「ファイル名とクラス名を教えて」。こうしてメンタルモデルを先に構築してから、実装フェーズに入る。この順番を守ると、バイブコードに逃げずに、ちゃんと自分で理解しながら PR が出せます。
